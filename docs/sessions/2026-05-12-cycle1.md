### Bloco 4 — Bug arquitetural: 4 agentes em paralelo no mesmo working tree

Tentativa: disparar F1-S01 + F1-S02 + F1-S10 + F0-S07 simultaneamente para acelerar.

**O que quebrou:**

- HEAD acabou em `feat/f1-s10` com arquivos de `f1-s01` no working tree.
- `feat/f1-s02` recebeu 2 commits de claim duplicados (race condition).
- Apenas F0-S07 sobreviveu intacto (último a finalizar).
- 3 dos 4 agentes foram rejeitados pelo usuário antes da degradação total.

**Causa raiz:** git só tem 1 working tree por repo. Agentes paralelos sem `isolation: "worktree"` fazem `git checkout` no mesmo disco e poluem o trabalho um do outro.

## Lições — Top 5

1. **`git worktree` ou sequencial. Nunca >1 agente no mesmo working tree.**
   Detalhe em PROTOCOL.md §7.1.

2. **STATUS.md é view derivada, não fonte da verdade.**
   Slot frontmatters são canônicos. `slot.py sync` re-renderiza. Editar à mão sempre causa drift entre branches.

3. **Scripts cortam ~80% das operações repetitivas.**
   `slot.py claim` substitui `git checkout main && pull && checkout -b ... && edit frontmatter && edit STATUS.md && commit`. De 5-7 comandos para 1 atômico.

4. **`Grep` em docs grandes economiza muito.**
   `docs/03`, `06`, `10`, `17`, `18` somam 2500+ linhas. Cada agente precisava de 1-2 seções. `Grep -A 40` em vez de `Read` inteiro = 10x menos tokens.

5. **Pre-flight é barato e salva muito.**
   `python scripts/slot.py preflight` em 1 segundo detecta working tree sujo, branch errado, main divergente. Catch precoce evita 90% dos bugs operacionais.
