## 1. O que muda neste deploy

- **2 migrations novas:** `0071` (5 views materializadas) + `0072` (permissões `reports:export`
  - `billing:read` p/ gestor_regional). Ambas aditivas, transacionais (rollback seguro se falhar).
- **2 dependências novas no backend:** `exceljs`, `pdfkit` (+ `@types/pdfkit`) — a imagem de prod
  precisa ser **reconstruída** (não basta restart).
- **Worker novo:** `reports-refresh` (refresh das MVs a cada 5 min) — precisa estar no processo de
  workers de prod. Gated por `dashboard.enabled` (já ON).
- **Frontend:** rota `/relatorios` deixa de ser placeholder.
- **Payload de auth** passa a incluir `city_scope_ids` (F23-S12) — usuários precisam **relogar**
  para o front receber o novo campo (o restart já força relogin).

## 5. Flags — ligar AMBAS no deploy (decisão travada), pós-smoke, sem novo restart

Fazer **após** o smoke (§3.5) passar. Caminho canônico: **UI admin de feature flags**
(`flags:manage`) — ligar:

- `reports.export.enabled`: OFF → **ON** (libera exportação CSV/XLSX/PDF).
- `dashboard.by_agent.enabled`: OFF → **ON** (quebra por agente).
- `dashboard.followup_metrics.enabled`: **deixar OFF** (Fase 6, não implementado).

Fallback via SQL (tabela `feature_flags`, coluna `status` enum `enabled|disabled|internal_only`):

```sql
UPDATE feature_flags SET status = 'enabled'
WHERE key IN ('reports.export.enabled', 'dashboard.by_agent.enabled');
```

> A página `/relatorios` e os relatórios de leitura **não dependem de flag** — ficam ativos já no
> deploy para quem tem permissão (efeito imediato do restart). As flags acima só liberam
> exportação e a quebra-por-agente. Mudança de flag é dinâmica (cache ~30s), sem restart.
